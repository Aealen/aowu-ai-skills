# 硬编码与通用性问题分析报告

> 审查范围：`scripts/build_docx.py`、`scripts/parse_pymupdf.py`
> 审查目标：找出所有违背"根据原 PDF 实际样式转换"原则的硬编码逻辑
> 日期：2026-07-13

---

## 一、字体名硬编码（3 处）

### 问题 1.1：`_FONT_MAP` 字体映射表全写死

**位置**：`build_docx.py:152-173`

**现状**：
```python
_FONT_MAP = {
    "simsun": "宋体", "song": "宋体", "songti": "宋体",
    "simhei": "黑体", "hei": "黑体", "heiti": "黑体",
    "fangsong": "仿宋", ...
}
```
PDF 中提取的字体名通过子串匹配映射到固定的中文字体名（"宋体"/"黑体"/"仿宋"/"楷体"/"微软雅黑"）。

**根因**：设计之初假设目标环境是中文 Windows 且安装了这五大基础字体。所有映射目标值都是中文 Windows 特有的字体名。

**通用性影响**：
- macOS/Linux 上字体名不同（Songti SC / Heiti SC），写入"宋体"后 Word 找不到字体
- 繁体/日文/韩文 PDF 的字体（MingLiU / Mincho / Batang）全部走兜底，保留 PDF 原始名，Word 也不识别
- 子串匹配过于激进：`"song" in name` 会误匹配 "Tsong"（藏文），`"kai" in name` 会匹配 "Kailash"
- 该映射表基于单一招标 PDF 样本（tuning-guide.md 明确标注），未经验证的其他字体家族无覆盖

**处理方案**：不再做字体名映射。直接保留 PDF 中 PyMuPDF 提取的原始字体名写入 DOCX。系统字体匹配由 Word/LibreOffice 渲染时自动回退处理，这是比人工映射表更通用的做法。

---

### 问题 1.2：加粗判断的字体名白名单（两处重复）

**位置**：
- `parse_pymupdf.py:64-71`：`_BOLD_FONT_PATTERNS = ("bold", "heavy", "black", "simhei", "stheiti")`
- `build_docx.py:346-350`：`_CID_BOLD_FONTS = {"simhei", "stheiti"}`

**现状**：`_is_bold()` 先检查 PyMuPDF flags 的 bit4，不中再检查字体名是否包含白名单关键词。两个文件各自维护一份名单，且内容不完全一致。

**根因**：CID 字体的 font flags bit4（粗体位）经常漏报，需要字体名兜底。但兜底名单是手工列举的，无法覆盖所有 CID 粗体字体。

**通用性影响**：
- tuning-guide.md 明确记录 FZXBSJW（方正小标宋）需要粗体推断，但两份名单都没包含它
- `"black"` 关键词会误匹配无关字体名（如 "Blackbird"）
- 两处名单重复维护，修改一处忘改另一处会产生不一致

**处理方案**：已实现的像素密度检测（`_check_cid_bold_by_density`）是更通用的方案，不依赖字体名。应将字体名白名单作为密度检测的补充而非独立路径，并消除两处重复。最终目标：加粗判断完全由 flags + 像素密度决定，不维护字体名名单。

## 二、正则匹配（2 处）

### 问题 2.1：`_looks_like_label` 用正则判断表格条款号

**位置**：`build_docx.py:998-1016`

**现状**：
```python
def _looks_like_label(text):
    if re.match(r"^\d+[\.\、]\d*", text): return True    # 1.1 / 1、
    if re.match(r"^第[一二三四五六七八九十百]", text): return True
    if len(text) <= 6 and not re.search(r"[。，；！？]", text): return True
    return False
```
该函数被 `_fix_crosspage_cell_misplacement` 调用，决定是否将表格 col0 的文本移到 col1。判断逻辑完全基于文本内容的正则/长度规则。

**根因**：跨页表格换页时 MinerU 会把正文内容错放到条款号列。为了区分"正常条款号"和"错位的正文"，用正则匹配编号格式 + 短文本规则来识别条款号。

**通用性影响**：
- 假设条款号只有"数字.数字"和"第X"两种格式，遗漏 ①②③、a)、（1）、一、等常见编号
- `len(text) <= 6` 把正常短内容（"是"/"否"/"通过"/"北京"/日期）误判为标签
- 中文数字正则只到"百"，漏"千/万/零"
- 这是一个**会修改表格内容**的函数，误判会直接破坏数据

**处理方案**：改为基于 bbox 坐标判断。PDF 原文中条款号列和内容列有固定的 x 坐标分界线（列边界）。如果 col0 的文本 x 坐标落在内容列的 x 范围内，就是错位——不需要分析文本内容。用 PyMuPDF 提取的单元格 bbox 坐标做空间判断，不依赖文本模式。

---

### 问题 2.2：标题层级正则是死代码

**位置**：`build_docx.py:36-42, 95-115`

**现状**：定义了 `_RE_CHAPTER`、`_RE_LEVEL1/2/3` 四个正则和 `_guess_heading_level()` 函数，但全文件无任何调用。标题层级已改为用 `_toc_level`（PDF 大纲）。

**根因**：早期版本用正则猜测标题层级，后来改为 PDF 大纲匹配，正则变为死代码但未清理。

**通用性影响**：不影响运行（死代码），但 tuning-guide.md 仍把正则描述为核心机制，构成误导。且正则本身只覆盖中文招标格式（"第X章"），对学术论文/英文文档无效。

**处理方案**：删除死代码（正则常量 + `_guess_heading_level` 函数），更新 tuning-guide.md。

---

## 三、写死默认值与魔法数字（8 处）

### 问题 3.1：图片宽度写死 `Inches(5.5)`

**位置**：`build_docx.py:1146`

**现状**：`doc.add_picture(str(full_path), width=Inches(5.5))`

**根因**：MVP 阶段简单处理，未从 PDF 提取图片实际尺寸。

**通用性影响**：所有图片统一 5.5 英寸宽，窄页溢出、小图标被放大、宽幅图表被压缩。

**处理方案**：从 PDF 中提取图片的实际 bbox 尺寸（pt→inch），按原始比例设置 DOCX 图片宽度。PyMuPDF 的 `page.get_image_bbox()` 或 block 的 bbox 可获取原始尺寸。

---

### 问题 3.2：A4 页面尺寸写死 `[595, 842]`

**位置**：`build_docx.py:246, 1330, 1331, 1453` 等共 6 处

**现状**：`page_size = page.get("page_size", [595, 842])`

**根因**：page_size 缺失时的兜底值。

**通用性影响**：只在 MinerU 数据无 page_size 时触发（实际几乎都有），风险低。但 Letter(612×792)、A3 等非 A4 文档如果恰好缺数据会出错。

**处理方案**：如果 page_size 缺失，从 PyMuPDF 的 `page.rect.width/height` 获取。不写死 A4。

---

### 问题 3.3：行高测量范围写死 `12-45pt`

**位置**：`build_docx.py:679`

**现状**：`if 12 <= gap <= 45:` — 行高测量时只接受 12-45pt 范围的 y 坐标差值。

**根因**：过滤异常值（如 block 间距被误算为行高）。12pt 是招标文件正文的常见行高下限，45pt 是上限。

**通用性影响**：8-9pt 脚注的行高被排除；48pt+ 封面大标题的行高被排除。对脚注密集或大字标题的 PDF 会丢失行高数据。

**处理方案**：范围改为相对值——基于该 block 内的字号计算合理行高范围（如 `size * 0.8` 到 `size * 3.5`），不写死绝对值。

---

### 问题 3.4：行距兜底值不统一（22pt / 24pt / 22.0pt）

**位置**：
- `build_docx.py:1084`：`_Pt(line_height or 22)` — 表格单元格行距
- `build_docx.py:1194`：`Pt(block.get("_line_height", 24))` — 目录行距
- `build_docx.py:1428`：`doc_median_lh = 22.0` — 文档行高中位数兜底

**根因**：不同时期各自写的兜底值，缺乏统一来源。

**通用性影响**：已有 `_doc_line_height`（文档级行高中位数）机制，但表格和目录没接上，仍用写死值。22pt 是招标 PDF（FangSong 16号 × 1.4）的经验值，小字号文档会被撑大。

**处理方案**：所有行距兜底统一改为 `block.get("_doc_line_height", ...)`。极端兜底值（当全文档无行高数据时）也可从主导字号估算（`dominant_size * 1.4`）。

---

### 问题 3.5：左边距兜底 `82`

**位置**：`build_docx.py:240, 277, 479`

**现状**：`doc_x0 = 82`、`block.get("_page_x0", 82)`

**根因**：`_detect_page_layout` 失败时的兜底值。82pt 是招标文件的经验左边距。

**通用性影响**：只在布局检测完全失败时触发，风险低。但不同版式的 PDF 左边距不同。

**处理方案**：兜底改为从 section margin 获取（`_detect_page_margins` 的结果），或从 MinerU 首页 block bbox 的最小 x0 推断。不写死 82。

---

### 问题 3.6：像素密度加粗阈值矛盾（1.3 vs 1.4）

**位置**：`parse_pymupdf.py:134（文档字符串）` vs `parse_pymupdf.py:220（代码）`

**现状**：文档字符串写"超过基线 1.4 倍"，代码用 `bl * 1.3`。

**根因**：调参时改了代码忘了改文档。

**处理方案**：统一为代码值 1.3，更新文档字符串。阈值本身是经验值，后续可考虑参数化。

---

### 问题 3.7：`font_baseline` 取最小字号作为正文基线

**位置**：`parse_pymupdf.py:191-194`

**现状**：
```python
size_densities.sort()
font_baseline[font_name] = size_densities[0][1]  # 取最小字号
```
注释说"最小字号通常是正文"，但实际脚注/页眉才是最小字号。

**根因**：错误假设最小字号 = 正文字号。

**通用性影响**：当某字号无同字号基线时，用字体级基线兜底。如果兜底取的是脚注密度（比正文低），会导致正常文字被误判为粗体。

**处理方案**：字体级基线改为取**出现次数最多的字号**（众数字号 = 正文），而非最小字号。

---

### 问题 3.8：HTML 实体清理不完整

**位置**：`build_docx.py:988-990`

**现状**：只清理 `&nbsp;` 和 `&amp;`，遗漏 `&lt;`、`&gt;`、`&quot;`、`&#39;` 等。

**根因**：只处理了当时遇到的两个实体，未做完整覆盖。

**通用性影响**：表格单元格含 `<`、`>`、引号时（如"投标额 < 100万"），DOCX 中出现 `&lt;` 字面字符串。

**处理方案**：改用 Python 标准库 `html.unescape()` 一步完成所有实体反转义。

---

## 四、其他

### 问题 4.1：加粗检测两处重复

**位置**：`parse_pymupdf.py:64-71`（提取层）+ `build_docx.py:346-350`（渲染层）

**现状**：提取层 `_is_bold()` 和渲染层 `_apply_style()` 各自维护一份 CID 加粗字体名单。提取层先判断，渲染层再补判。

**根因**：渲染层的补判是为了处理"未重新提取 spans 就直接 build"的场景（旧 spans 数据没有密度检测结果）。

**处理方案**：加粗判断统一在提取层（`parse_pymupdf.py`）完成，结果存入 `_style.bold`。渲染层只消费结果，不做二次判断。
